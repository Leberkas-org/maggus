# Bug: Discord presence stuck on "In Main Menu" / "Watching for changes" during interactive prompt sessions

## Summary

When launching an interactive Claude Code session via the main menu's "prompt" command, Discord Rich Presence never updates to the expected skill verb (e.g., "Planning", "Reporting Bug"). It stays on either "In Main Menu" (from the menu loop) or "Watching for changes" (from the daemon). Multiple independent IPC connections fight over the same Discord Application ID, and the interactive session's connection always loses.

## Related

- **Commit:** 8665087 (feat(discord): add skill-to-verb mapping and wire prompt command to Discord presence)
- **Commit:** 888cffd (fix(discord): add session lock to coordinate presence between prompt and daemon)

## Steps to Reproduce

1. Start the daemon (`maggus daemon` or have it running in background)
2. Run `maggus` to open the main menu
3. Select "prompt" → pick any skill (e.g., `/maggus-plan`)
4. Check Discord Rich Presence
5. Observe: status shows "In Main Menu" or "Watching for changes", never "Planning"

## Expected Behavior

Discord presence should show the selected skill's verb (e.g., "Planning", "Reporting Bug", "Consulting") for the duration of the interactive Claude Code session. The daemon should not touch Discord presence at all. Whichever interactive UI updated last should win.

## Root Cause

There are four independent `Presence` instances that each open their own Discord IPC connection with the same Application ID:

| Location | State shown |
|---|---|
| `src/cmd/root.go:61-85` | "In Main Menu" |
| `src/cmd/prompt.go:103-126` | Skill verb (e.g., "Planning") |
| `src/cmd/daemon_keepalive.go:71-74` | "Watching for changes" |
| `src/cmd/work.go:230-233` | Task details ("Working on...") |

Discord's IPC pipe (`\\.\pipe\discord-ipc-0` on Windows) accepts multiple client connections, but only one connection per Application ID effectively controls the displayed presence. The connections established by root.go (at startup) and the daemon (at daemon start) are long-lived and overwrite any updates from the prompt command's shorter-lived connection.

### Problem 1: Root menu keeps its own Presence connection alive

`src/cmd/root.go:61-85` creates a `Presence` and keeps it connected for the entire application lifetime. When the user selects "prompt", `runPrompt()` is called at line 131 via `sub.RunE()`. Inside `runPrompt()` (`src/cmd/prompt.go:103-126`), a **second** `Presence` is created and connected.

The root menu's connection was established first and remains active. When `runPrompt()` returns, the menu loop iterates back to lines 88-100 and immediately overwrites with "In Main Menu". Even during the prompt session, the root menu's connection may take precedence.

### Problem 2: Daemon should not manage presence at all

The daemon (`src/cmd/daemon_keepalive.go:71-74, 129-134`) creates its own `Presence` and updates to "Watching for changes". The existing `sesslock.IsActive()` guard (line 129) is insufficient — it only prevents some updates, and the daemon's long-lived connection still competes with interactive sessions. Per user preference, the daemon should never touch Discord presence.

### Problem 3: Work command creates yet another connection

`src/cmd/work.go:230-233` creates its own `Presence` when invoked. When launched from the menu, this is a third competing connection alongside root.go's. The work command's presence updates (task details at `work_task.go:109-120`) may be overwritten by the menu's "In Main Menu" when work finishes.

### Core architectural issue

Each entry point independently creates and manages its own `Presence` connection. The fix requires:
1. A single shared `Presence` instance owned by the root menu, passed to subcommands
2. Complete removal of Discord presence from the daemon
3. "Last update wins" semantics — no coordination locks needed, just one connection

## User Stories

### BUG-003-001: Remove Discord presence from daemon

**Description:** As a user, I don't want the background daemon to control my Discord status, so that only interactive UIs (menu, prompt, work) determine what's shown.

**Acceptance Criteria:**
- [x] `src/cmd/daemon_keepalive.go` no longer creates a `discord.Presence` instance
- [x] `src/cmd/daemon_keepalive.go` no longer calls `presence.Update()` with "Watching for changes"
- [x] The `presence` parameter is removed from `runOneDaemonCycle()` and `runKeepalive()`
- [x] `src/cmd/work_task.go` session lock checks (`sesslock.IsActive`) for presence can be removed (no daemon competing)
- [x] The `sesslock` package usage in prompt.go can be removed (no daemon to coordinate with)
- [x] Daemon still functions correctly for its primary job (watching files, running work cycles)
- [x] No regression in daemon file watching or task execution
- [x] `go vet ./...` and `go test ./...` pass

### BUG-003-002: Share a single Presence connection from root menu to subcommands

**Description:** As a user, I want a single Discord IPC connection shared across the menu, prompt, and work commands, so that the last UI to update always wins and my Discord status accurately reflects what I'm doing.

**Acceptance Criteria:**
- [ ] Root menu (`src/cmd/root.go`) creates a single `Presence` instance and passes it to subcommands
- [ ] `runPrompt()` accepts and uses the shared `Presence` instead of creating its own; when run standalone (`maggus prompt`), it creates its own
- [ ] `runWork()` / work command accepts and uses the shared `Presence` instead of creating its own; when run standalone (`maggus work`), it creates its own
- [ ] Prompt command updates presence to the skill verb ("Planning", "Reporting Bug", etc.) on the shared connection
- [ ] Work command updates presence to task details on the shared connection
- [ ] When a subcommand returns, root menu updates presence back to "In Main Menu" on the same connection
- [ ] Discord shows the correct status for the entire duration of each subcommand
- [ ] No regression: standalone `maggus prompt` and `maggus work` still manage their own presence
- [ ] `go vet ./...` and `go test ./...` pass

> **Note:** BUG-003-001 should be implemented first (removes daemon competition), then BUG-003-002 (consolidates interactive UIs onto one connection).
