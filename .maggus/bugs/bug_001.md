# Bug: Tree shows blank after toggling completed items visibility

## Summary

Toggling the "show all" / hide-completed items view (`alt+a`) while scrolled down in the left pane tree leaves `treeScrollOffset` stale. After the plan list shrinks (completed items hidden, or an expanded plan collapsed), the stale offset points at or past the end of the new item list, causing `renderLeftPane` to slice an empty window and render no items.

## Steps to Reproduce

1. Open `maggus status` with several features/bugs, some completed
2. Expand a few plans so the tree is taller than the screen
3. Scroll down so `treeScrollOffset > 0`
4. Press `alt+a` to hide completed items — the plan list shrinks
5. Observe: the left pane shows no items (blank or "No features found")

Also reproducible by:
- Scrolling to the bottom, then collapsing an expanded plan with many tasks

## Expected Behavior

After toggling visibility or collapsing a plan, the tree should re-clamp the scroll offset to a valid position and continue showing the remaining visible items.

## Root Cause

**Primary cause — `alt+a` handler does not clamp scroll after reload**

`status_update.go` lines 586–596:

```go
case "alt+a":
    m.showAll = !m.showAll
    plans, a, err := loadPlansWithApprovals(...)
    ...
    m.rebuildForSelectedPlan()
    return m, nil   // ← clampTreeScroll() never called
```

`reloadPlans()` (`status_model.go:266`) correctly resets `m.treeScrollOffset = 0` before rebuilding, but the `alt+a` handler skips this entirely. When the plan list shrinks, `treeScrollOffset` is left pointing past the end of the new item list.

**Secondary cause — off-by-one guard in `renderLeftPane`**

`status_leftpane.go` lines 69–77:

```go
scrollOff := m.treeScrollOffset
if scrollOff > len(allItems) {
    scrollOff = len(allItems)   // ← clamps to len, not len-1
}
end := scrollOff + availH
if end > len(allItems) {
    end = len(allItems)
}
items := allItems[scrollOff:end]   // ← empty slice when scrollOff == len(allItems)
```

When `scrollOff` is clamped to exactly `len(allItems)`, the resulting slice is always empty regardless of how many items exist. The guard should clamp to `max(0, len(allItems)-1)`.

## User Stories

### BUG-001-001: Call clampTreeScroll after alt+a plan reload

**Description:** As a user, I want the tree to remain visible after toggling completed item visibility so I can continue navigating.

**Acceptance Criteria:**
- [x] In the `alt+a` key handler (`status_update.go` ~line 595), add `m.clampTreeScroll()` after `m.rebuildForSelectedPlan()` and before `return m, nil`
- [x] After pressing `alt+a` while scrolled down, the tree shows the items nearest to the previous scroll position (or the end of the list if the list is now shorter)
- [x] No regression in `alt+a` toggle behavior (completed items correctly shown/hidden)
- [x] `go vet ./...` and `go test ./...` pass

### BUG-001-002: Fix off-by-one scroll guard in renderLeftPane

**Description:** As a developer, I want the scroll guard in `renderLeftPane` to be safe against edge cases so the tree never renders blank when items exist.

**Acceptance Criteria:**
- [ ] In `renderLeftPane` (`status_leftpane.go` ~line 70), change the guard from `if scrollOff > len(allItems) { scrollOff = len(allItems) }` to `scrollOff = min(scrollOff, max(0, len(allItems)-1))` — ensuring `scrollOff` never equals `len(allItems)` when items exist
- [ ] When `len(allItems) == 0`, `scrollOff` is clamped to 0 (no panic on empty slice)
- [ ] Tree never renders blank when `allItems` is non-empty, regardless of `treeScrollOffset` value
- [ ] `go vet ./...` and `go test ./...` pass
