# Bug: Menu startup blocks ~3s on Discord IPC connection when daemon is running

## Summary

Starting `maggus` (the menu) takes about 3 seconds when Discord presence is enabled and a daemon is running in the background. The Discord IPC connection in `runMenu` blocks synchronously before the TUI renders, and on Windows the named pipe `OpenFile` call has no timeout — unlike Unix which has a 5-second `DialTimeout`.

## Related

- **Commit:** 2c61fbf (fix(discord): implement bounded-time reads for Windows named pipe IPC)

## Steps to Reproduce

1. Enable `discord_presence: true` in `.maggus/config.yml`
2. Start the daemon with `maggus start`
3. Run `maggus` to open the menu
4. Observe: ~3 second delay before the menu appears

## Expected Behavior

The menu should appear instantly. Discord presence connection should happen in the background without blocking the TUI.

## Root Cause

Two factors combine to cause the delay:

### 1. Windows named pipe open has no timeout (`ipc_windows.go:14`)

```go
func connectIPC() (net.Conn, error) {
    f, err := os.OpenFile(`\\.\pipe\discord-ipc-0`, os.O_RDWR, 0)
```

Unlike the Unix implementation (`ipc_unix.go:24`) which uses `net.DialTimeout("unix", socket, 5*time.Second)`, the Windows implementation calls `os.OpenFile` with no timeout. When Discord is busy (e.g. the daemon already holds a connection and Discord is processing), this call can block for several seconds waiting for the pipe to become available.

### 2. Discord connection is synchronous in the menu startup path (`root.go:62-65`)

```go
if cfg, err := config.Load(cwd); err == nil && cfg.DiscordPresence {
    presence = &discord.Presence{}
    _ = presence.Connect()  // blocks until IPC + handshake complete
}
```

The `Connect()` call at `root.go:64` runs synchronously before the TUI is created at line 81. This means the entire startup pipeline — `OpenFile` on the pipe + handshake write + handshake read (5s timeout) — must complete before the user sees anything.

### 3. Metrics lock contention is a minor contributor (`root.go:44`)

The `PersistentPreRun` at `root.go:43-47` calls `IncrementMetrics` which acquires a file lock (`metrics.lock`). If the daemon holds this lock, the menu retries up to 10 times with 50ms delays (max 450ms). This is a secondary contributor on top of the Discord delay.

## User Stories

### BUG-001-001: Move Discord presence connection to a background goroutine in menu startup

**Description:** As a user, I want the menu to appear instantly so that I don't have to wait for Discord IPC to connect before I can interact with maggus.

**Acceptance Criteria:**
- [x] `runMenu` in `root.go` launches Discord `Connect()` in a goroutine instead of blocking
- [x] The TUI renders immediately, regardless of Discord connection status
- [x] Discord presence updates are sent once connection completes (no updates lost)
- [x] If Discord connection fails or times out, the menu still works normally
- [x] No race conditions on the `presence` variable (mutex or channel-based synchronization)
- [x] No regression in Discord presence for daemon mode or work command
- [x] `go vet ./...` and `go test ./...` pass

### BUG-001-002: Add a connection timeout to Windows named pipe IPC

**Description:** As a user on Windows, I want the Discord IPC connection to fail fast when Discord is unresponsive so that startup is never blocked indefinitely.

**Acceptance Criteria:**
- [ ] `connectIPC()` in `ipc_windows.go` enforces a timeout (e.g. 2-3 seconds) on the `OpenFile` call
- [ ] Timeout matches the spirit of the Unix implementation's 5-second `DialTimeout`
- [ ] If the pipe open times out, `connectIPC` returns `nil, nil` (Discord not available)
- [ ] The goroutine-based timeout cleans up properly (no leaked file handles or goroutines)
- [ ] No regression in Discord presence when Discord is running and responsive
- [ ] `go vet ./...` and `go test ./...` pass
