# Bug: Discord Rich Presence not cleared after all Maggus instances exit

## Summary

After closing all Maggus instances (including daemons), the Discord Rich Presence still shows Maggus as active. Discord should clear the activity when all connections close, but the current implementation has two issues: the `Close()` method doesn't read Discord's response after sending the clear-activity command (leaving the protocol out of sync), and the daemon code path never connects to Discord at all — so there's no presence to clear on daemon shutdown.

## Steps to Reproduce

1. Enable `discord_presence: true` in `.maggus/config.yml`
2. Run `maggus work` or `maggus` (menu) — Discord shows Maggus activity
3. Close the Maggus instance (Ctrl+C or menu quit)
4. Observe: Discord still shows Maggus Rich Presence as active

Alternative path with daemon:
1. Run `maggus start` (daemon mode)
2. Observe: Discord never shows presence (daemon doesn't connect)
3. Run `maggus stop`
4. If a non-daemon instance was open earlier: presence may still be stuck

## Expected Behavior

When all Maggus instances exit, Discord should clear the Rich Presence immediately. The activity should disappear from the user's Discord profile.

## Root Cause

There are two independent issues:

### Issue 1: `Close()` doesn't read the response after clearing activity

In `src/internal/discord/discord.go:117-121`, the `Close()` method sends a `buildClearActivityPayload` but never reads the response:

```go
if !p.disconnected {
    payload := buildClearActivityPayload(os.Getpid())
    _ = writeMessage(p.conn, opFrame, payload)
}
```

The `Update()` method (line 97-101) always reads the response to keep the protocol in sync. The `Close()` method skips this. Discord's IPC protocol expects the client to consume the response. When the connection is closed without reading the response, Discord may not process the clear-activity command, leaving the presence stuck.

Additionally, `Close()` does not send an `opClose` (opcode 2) message before closing the connection. The Discord IPC protocol defines opcode 2 as the close opcode, and sending it tells Discord explicitly that the client is disconnecting. Without it, Discord treats the connection drop as unexpected and may retain the activity for a timeout period.

### Issue 2: Daemon never initializes Discord presence

In `src/cmd/daemon_keepalive.go`, the `runOneDaemonCycle()` function (line 170) builds a `taskContext` at line 264-278 but never sets the `presence` field — it remains `nil`. The daemon's `runDaemonLoop()` function (line 24) also has no Discord initialization. This means daemon mode never shows or clears presence.
