# Bug: Menu auto-dispatches work every time it opens, causing an infinite work loop

## Summary

After the `auto_work` removal, the menu immediately dispatches work on every open when workable tasks exist — including when returning from a completed work run. This creates an infinite loop: menu opens → work dispatched → work finishes → menu re-opens → work dispatched again, making the menu completely unusable.

## Related

- **Commit:** f3a18d6 (fix(menu): remove auto_work setting and always auto-dispatch work)
- **Commit:** 98bf35d (fix(menu): auto-start work on menu launch when workable tasks exist)

## Steps to Reproduce

1. Have one or more workable tasks in any feature or bug file
2. Run `maggus` (launches the menu)
3. Observe: work is immediately dispatched without user interaction
4. Work runs and finishes (or errors), returning to the menu loop in `root.go`
5. Menu re-opens → workable tasks still exist → work is immediately dispatched again
6. Repeat indefinitely — the menu is never interactive

## Expected Behavior

The menu should display, allow the user to interact, and only auto-dispatch work when a **new** file appears (file watcher `HasNewFile` event). Returning to the menu after a work run should show the menu normally, not immediately re-dispatch work.

## Root Cause

`Init()` in `src/cmd/menu.go:407` unconditionally appends a `startupAutoWorkMsg` command whenever `m.summary.workable + m.summary.bugWorkable > 0`:

```go
// src/cmd/menu.go:422-427
if m.summary.workable+m.summary.bugWorkable > 0 {
    cmds = append(cmds, func() tea.Msg {
        return startupAutoWorkMsg{}
    })
}
```

When `startupAutoWorkMsg` is handled (line 463), it sets `selected = "work"` and returns `tea.Quit`, immediately exiting the menu into work mode.

The outer loop in `src/cmd/root.go:106` re-creates `newMenuModel(loadFeatureSummary())` after every work run and calls `p.Run()` again. Since tasks are still present after each run, `Init()` fires `startupAutoWorkMsg` every iteration.

The original intent was "auto-dispatch only on fresh startup before the user has interacted", but there is no flag distinguishing "first launch" from "returning from a work run". Every loop iteration looks identical to the menu.
