# Bug: Status view shows plain text log instead of rich live work TUI when daemon is running

## Summary

When the daemon is running, `maggus status` (and the menu "status" entry) shows a plain JSONL text log panel instead of the rich live TUI that `maggus work` used to render. The rework that unified status/log/work into a single menu entry (`eec6ced`) replaced the rich work view with a plain log reader, but never wired the `runner.TUIModel` (spinner, 3-zone progress, tool list, token stats) into the status command's daemon-attach mode.

## Related

- **Commit:** `eec6ced` (feat(tui): unify status/log/work into single menu entry with Tab toggle)
- **Commit:** `35102c5` (feat(menu): add attach menu item to watch daemon live log)

## Steps to Reproduce

1. Start the daemon: `maggus start` (or via menu)
2. Open `maggus status` (or select "status" from the menu)
3. Press `Tab` to switch to the live log panel
4. Observe: a plain text log (JSONL lines from `run.log`) is shown

## Expected Behavior

The status view's live mode should render the same rich TUI that `maggus work` used to show: spinner + status, 3-zone progress layout (top: status/task/output, middle: scrollable tool list, bottom: model/tokens/elapsed), not a plain text log reader.

## Root Cause

The "attach to daemon" feature was implemented as a JSONL file reader in `src/cmd/status_runlog.go`. It polls `run.log` every 200ms and renders lines using `formatLogLine()`. This is fundamentally different from the live `runner.TUIModel` which receives structured messages (tool events, output, usage) in real-time via bubbletea.

The daemon runs using `nullTUIModel` (`src/cmd/daemon_tui.go`) — a headless model that discards all display events. It writes to `run.log` via the `runlog` package, but the structured state (spinner frame, tool entries, token accumulation, scroll state) is never exported to a form the status view can consume.

To render the rich TUI from status, one of these approaches is needed:
- **Option A (recommended):** The daemon writes a structured "state snapshot" file (e.g. `.maggus/runs/<RUN_ID>/state.json`) alongside `run.log`, containing the current tool list, token totals, task ID/title, status string, and model breakdown. The status view reads and reconstructs a display-only `TUIModel` from these snapshots (read-only, no stop-picker or interactive controls).
- **Option B:** IPC via a named pipe or unix domain socket from the daemon to any attached status viewer.

Option A is simpler and consistent with the existing file-based approach.

## User Stories

### BUG-006-001: Write structured state snapshot from daemon work loop

**Description:** As a developer, I want the daemon to write a `state.json` snapshot file into its run directory on each tool event, task start, and task completion, so that the status view can read live state without IPC.

**Acceptance Criteria:**
- [x] A `StateSnapshot` struct is defined (e.g. in `internal/runlog`) with fields: `TaskID`, `TaskTitle`, `FeatureFile`, `Status`, `ToolEntries` (type, icon, description, timestamp), `TokenInput`, `TokenOutput`, `TokenCost`, `ModelBreakdown` (map[string]ModelTokens), `Commits` ([]string), `UpdatedAt` (RFC3339)
- [x] The daemon work loop (via `nullTUIModel` or a new callback) writes/overwrites `.maggus/runs/<RUN_ID>/state.json` on each `agent.ToolMsg`, `agent.StatusMsg`, `agent.UsageMsg`, `IterationStartMsg`, and `CommitMsg`
- [x] The file is valid JSON and is written atomically (write to temp file, rename)
- [x] The file is removed or zeroed when the daemon exits cleanly
- [x] `go vet ./...` and `go test ./...` pass

### BUG-006-002: Replace log panel in status view with rich live TUI renderer

**Description:** As a user, I want the Tab panel in `maggus status` (when the daemon is running) to render the same rich live view as the old work TUI — with spinner, tool list, and token stats — instead of the plain JSONL text log.

**Acceptance Criteria:**
- [ ] When `m.showLog` is true and the daemon is running, `viewLog()` in `src/cmd/status.go` reads and parses the latest `state.json` snapshot from the active run directory

- [ ] The view falls back to the plain JSONL log reader if no `state.json` exists (daemon not writing snapshots or old daemon version)
- [ ] When the daemon is not running, the panel continues to show the last log lines as before
- [ ] `Tab` continues to toggle between the feature list and the live panel
- [ ] Scroll (j/k/up/down), auto-scroll to bottom (G), and jump to top (g) work on the tool list in the new panel
- [ ] The animated spinner only ticks while the daemon is actively running (not polling when idle)
- [ ] No regression in the feature list / tab bar / task navigation in the status view
- [ ] `go vet ./...` and `go test ./...` pass
