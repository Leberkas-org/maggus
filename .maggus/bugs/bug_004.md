# Bug: Startup auto-dispatches work view when approved tasks exist, bypassing the menu

## Summary

When maggus starts and there are workable tasks (including approved bugs/features), it immediately bypasses the main menu and auto-launches the `work` command TUI — without any user intent. This happens unconditionally on first launch when no daemon is running, which is wrong. The user should always see the menu and decide when to start work.

## Steps to Reproduce

1. Have a `.maggus/` project with at least one approved feature or bug file containing a workable task
2. Ensure no daemon is running
3. Run `maggus` (no subcommand)
4. Observe: the work TUI opens immediately and begins processing the task — the main menu is never shown

## Expected Behavior

`maggus` should always show the main menu on startup, regardless of whether workable tasks exist. The user explicitly chooses to start work (via daemon, status view, or the work command).

## Root Cause

Two related places in `menu_update.go` unconditionally auto-dispatch the `work` command:

**1. Startup auto-dispatch** (`menu_update.go:136-140` + `177-181`):

```go
// Init()
if m.firstLaunch && m.summary.workable+m.summary.bugWorkable > 0 {
    cmds = append(cmds, func() tea.Msg { return startupAutoWorkMsg{} })
}

// Update()
case startupAutoWorkMsg:
    m.selected = "work"
    m.args = []string{"--count", "0"}
    return m, tea.Quit
```

On the very first menu open, if any workable tasks exist, `startupAutoWorkMsg` is fired and immediately exits the menu into `work --count 0`, skipping user interaction entirely.

**2. File-watcher auto-dispatch** (`menu_update.go:182-190`):

```go
case featureSummaryUpdateMsg:
    m.summary = loadFeatureSummary()
    workable := m.summary.workable + m.summary.bugWorkable
    if msg.HasNewFile && workable > 0 {
        m.selected = "work"
        m.args = []string{"--count", "0"}
        return m, tea.Quit
    }
```

When a new feature/bug file is created while the menu is open, the watcher also auto-dispatches to `work`.

**3. Approval status ignored** (`menu_model.go:126-165`):

`loadFeatureSummary()` counts all workable tasks without checking whether their parent plan is approved. This means even unapproved features contribute to the `workable` count that triggers the auto-dispatch.

## User Stories

### BUG-004-001: Remove startup auto-work dispatch

**Description:** As a user, I want `maggus` to always show the main menu on startup so I can decide when and how to start work, rather than having it auto-launch the work view.

**Acceptance Criteria:**
- [x] Remove the `startupAutoWorkMsg` type and all references to it from `menu_update.go` and `menu_model.go`
- [x] Remove the `firstLaunch` guard block in `Init()` that appends `startupAutoWorkMsg` (`menu_update.go:136-140`)
- [x] The `firstLaunch` field on `menuModel` may be removed if no other code uses it after this change — verify before removing
- [x] Remove the `featureSummaryUpdateMsg` auto-dispatch block that fires `work` when a new file is created (`menu_update.go:184-190`); the summary should still reload, just not auto-dispatch
- [x] Starting `maggus` with approved workable tasks and no daemon shows the main menu normally
- [x] Starting `maggus` with no workable tasks still shows the main menu
- [x] No regression in daemon auto-start behavior (`autoStartDaemon` in `Init()` must remain unchanged)
- [x] `go vet ./...` and `go test ./...` pass

### BUG-004-002: Fix loadFeatureSummary to respect approval status

**Description:** As a user, I want the menu's workable task count to only reflect approved plans so that unapproved features do not influence menu behavior.

**Acceptance Criteria:**
- [ ] `loadFeatureSummary()` in `menu_model.go` filters plans by approval status before counting workable tasks — use the same approval check as `buildApprovedPlans()` in `work_loop.go`
- [ ] The `workable` and `bugWorkable` counts in `featureSummary` only include tasks from approved plans
- [ ] The menu header summary line accurately reflects only approved open tasks
